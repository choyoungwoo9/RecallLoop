# Recall Loop 현재 구현 기준 PRD

## 0. 문서 목적

이 문서는 `/FE/src`, `/BE/src`, 설정 파일, 기존 문서를 기준으로 현재 저장소에 실제로 구현되어 있는 기능을 PRD 형식으로 재구성한 문서다.

- 기준 시점: 현재 코드베이스
- 기준 원칙: 코드에 존재하는 기능만 서술하고, 아직 구현되지 않은 기능은 목표 기능으로 쓰지 않는다
- 표기 원칙:
  - `구현됨`: 현재 UI 또는 API로 동작 확인 가능한 기능
  - `내부 구현`: 코드에는 있으나 현재 UI에서 직접 노출되지 않는 기능
  - `제약`: 현재 구현상 한계 또는 의도된 제한

---

## 1. 제품 개요

### 1.1 제품명

Recall Loop

### 1.2 한 줄 설명

사용자가 학습 내용을 기록하면 AI가 문제를 생성하고, 생성된 문제를 하나의 전역 Queue에서 반복 순환하며 풀게 해주는 학습 복습 앱.

### 1.3 핵심 가치

- 학습 내용을 바로 문제화한다
- 문제를 직접 고르지 않고 순환 Queue로 반복 복습한다
- 한 기록을 모두 푼 시점에 원문과 내 답을 나란히 비교하게 해준다
- 사이클 단위로 학습 이력을 누적하고, 사용자 난이도 평가를 다음 루프의 난이도 조정에 반영한다

### 1.4 현재 구현 범위

- 단일 사용자 전제의 로컬/개인용 학습 앱
- 학습 기록 생성, 조회, 삭제
- 학습 기록별 문제 생성 설정 생성/조회/삭제
- Gemini 기반 문제 생성
- 전역 Queue 기반 문제 풀이
- 풀이 시간 기록
- 학습 기록 단위 완주 요약 및 난이도 평가
- 사이클 완료 후 히스토리 이관
- 대시보드 및 풀이 기록 조회

### 1.5 현재 구현에서 제외된 범위

- 회원가입, 로그인, 멀티 유저
- 학습 기록 수정
- 문제 수정
- 정답 자동 채점
- 검색, 필터, 정렬 고도화
- 수동 문제 순서 변경
- 여러 Queue 또는 과목별 Queue
- 관리자 화면
- 테스트 자동화 커버리지

## 2. 대상 사용자

### 2.1 주요 사용자

학습 내용을 텍스트로 정리한 뒤, 그것을 기반으로 반복 복습하고 싶은 개인 학습자.

### 2.2 사용자 특성

- 강의, 책, 블로그, 문서 내용을 복사해 학습 기록으로 저장할 수 있다
- 객관식보다 서술형/회상형 복습을 선호한다
- 문제를 직접 선별하기보다 시스템이 다음 문제를 정해주길 원한다
- 정답률보다 "스스로 회상했는지"와 "이 문제가 적절했는지"를 중요하게 본다

## 3. 제품 목표

### 3.1 비즈니스/제품 목표

- 학습 기록을 복습 가능한 문제 세트로 빠르게 전환한다
- 사용자가 다음에 무엇을 풀지 고민하지 않게 한다
- 반복 학습의 마찰을 줄이고 루프를 유지하게 한다
- 학습 품질 피드백을 다음 사이클 문제 난이도에 반영한다

### 3.2 사용자 목표

- 배운 내용을 기록하고 잊기 전에 복습하고 싶다
- 한 번 만든 학습 재료를 여러 번 돌려보고 싶다
- 내 답변과 원문/정답을 비교하고 싶다
- 어떤 기록을 얼마나 반복했는지 확인하고 싶다

## 5. 핵심 사용자 여정

### 5.1 기본 여정

1. 사용자는 대시보드 또는 학습 기록 목록으로 진입한다.
2. 새 학습 기록을 작성한다.
3. 특정 학습 기록 상세에서 문제 생성 설정을 만든다.
4. 설정 기준으로 AI 문제를 생성한다.
5. Queue 현황 또는 문제 풀이 화면으로 이동한다.
6. 전역 Queue가 제시하는 현재 문제를 풀고 답을 제출한다.
7. 특정 학습 기록의 모든 활성 문제를 현재 사이클에서 다 풀면 완주 요약 모달이 열린다.
8. 사용자는 난이도 평가를 반드시 저장한 뒤 계속 풀이를 진행할 수 있다.
9. 전체 Queue를 한 바퀴 다 돌면 사이클 완료 전환 애니메이션이 재생되고, 현재 사이클의 풀이들이 히스토리로 이관된다.
10. 사용자는 대시보드와 풀이 기록에서 누적 학습 현황을 확인한다.

### 5.2 삭제 여정

1. 사용자는 학습 기록 상세에서 기록을 삭제할 수 있다.
2. 삭제 시 학습 기록 자체는 제거된다.
3. 연관 문제는 큐에서 비활성화되고 학습 기록/설정 참조가 해제된다.
4. 기존 풀이 기록은 보존되며, 이후 화면에서는 제목이 `(삭제된 기록)`으로 표기될 수 있다.

## 6. 기능 요구사항

### 6.1 대시보드

#### 목적

사용자가 현재 학습 상태와 누적 복습 상태를 한 화면에서 이해할 수 있게 한다.

#### 구현 요구사항

- 시스템은 대시보드 첫 진입 시 `/api/dashboard`를 호출한다.
- 시스템은 다음 개요 지표를 노출한다.
  - 총 루프 횟수
  - 총 학습 시간
  - 오늘 푼 문제 수
  - 오늘 학습 시간
  - 총 풀이 수
  - 고유 문제 수
  - 활성 문제 수
  - 학습 기록 수
  - 평균 풀이 시간
  - 연속 학습일
- 시스템은 현재 사이클 진행도를 원형 시각화와 텍스트로 함께 보여준다.
- 시스템은 현재 Queue의 다음 문제를 보여준다.
- 시스템은 최근 7일 활동량을 일자별 막대 차트로 보여준다.
- 시스템은 난이도 평가 분포와 문제 품질 피드백 누적값을 보여준다.
- 시스템은 현재 활성 Queue의 난이도 분포를 쉬움/보통/어려움으로 나누어 보여준다.
- 시스템은 가장 많이 푼 학습 기록 상위 5개를 보여준다.
- 사용자는 대시보드에서 바로 `문제 풀기`, `학습 기록 보기`, `새 학습 기록 작성`으로 이동할 수 있다.

#### 제약

- 대시보드 통계는 현재 사이클의 `quiz_attempt`와 과거 사이클의 `quiz_attempt_history`를 합산해 계산한다.
- 연속 학습일 계산은 `Asia/Seoul` 기준 날짜를 사용한다.

### 6.2 학습 기록 관리

#### 목적

학습 재료를 저장하고 문제 생성의 기반 문서를 관리하게 한다.

#### 구현 요구사항

- 사용자는 제목과 내용을 입력해 학습 기록을 생성할 수 있다.
- 제목 또는 내용이 비어 있으면 프론트엔드는 저장을 막고 오류 메시지를 보여준다.
- 사용자는 학습 기록 목록에서 모든 기록을 카드 형태로 볼 수 있다.
- 각 카드는 제목, 내용 일부, 생성일, 활성 문제 수를 보여준다.
- 사용자는 카드를 클릭해 상세 화면으로 이동할 수 있다.
- 상세 화면은 다음 탭을 제공한다.
  - 학습 내용
  - 생성된 문제
  - 풀이 기록
  - 문제 생성 설정
- 사용자는 학습 기록을 삭제할 수 있다.

#### 삭제 시 비즈니스 규칙

- 학습 기록이 삭제되면 연관 활성 문제는 Queue에서 제외된다.
- 문제 엔티티 자체는 남겨서 기존 풀이 기록의 FK 정합성을 보존한다.
- 연관 설정은 삭제된다.
- Queue 상태는 활성 문제 기준으로 다시 동기화된다.

#### 제약

- 학습 기록 수정 기능은 없다.
- 학습 기록 검색/필터 기능은 없다.

### 6.3 문제 생성 설정

#### 목적

같은 학습 기록에 대해 서로 다른 문제 생성 의도를 저장하게 한다.

#### 구현 요구사항

- 사용자는 하나의 학습 기록에 여러 문제 생성 설정을 만들 수 있다.
- 각 설정은 다음 정보를 가진다.
  - 설명
  - 생성할 문제 개수
  - 생성 시각
- 사용자는 설정 목록을 확인할 수 있다.
- 사용자는 설정을 삭제할 수 있다.
- 설정 생성 폼의 기본값은 다음과 같다.
  - 설명: `핵심 개념 정리`
  - 문제 개수: `5`
- 문제 개수 입력값은 프론트에서 최소 1 이상으로 보정된다.

#### 제약

- 설정 수정 기능은 없다.
- UI상 최대값은 20으로 보이지만, 백엔드 검증은 별도로 강제하지 않는다.

### 6.4 AI 문제 생성

#### 목적

학습 기록 원문을 기반으로 복습 가능한 문제를 자동 생성한다.

#### 구현 요구사항

- 사용자는 설정 카드에서 `문제 생성`을 눌러 AI 문제 생성을 요청할 수 있다.
- 시스템은 학습 내용, 설정 설명, 문제 수를 Gemini 프롬프트에 포함해 요청한다.
- 시스템은 정확히 지정한 개수의 JSON 배열 형식 응답을 기대한다.
- 응답은 문제와 정답 쌍으로 저장된다.
- 새로 생성된 문제는 전체 활성 Queue의 순서를 다시 정렬한 뒤 포함된다.
- 생성 완료 후 사용자는 성공 모달을 보고 `생성된 문제` 탭으로 이동할 수 있다.

#### 데이터 규칙

- 각 문제는 `question`, `answer`, `queueOrder`, `difficulty`, `originalQuizId`, `isActiveInQueue`를 가진다.
- 기본 난이도는 5다.
- 일반 생성 문제는 활성 Queue에 바로 포함된다.

#### 제약

- 생성 실패 시 사용자는 오류 모달만 본다.
- 중복 문제 제거, 품질 검수, 후편집 기능은 없다.

### 6.5 생성된 문제 조회 및 삭제

#### 목적

사용자가 현재 학습 기록에서 만들어진 문제를 확인하고 정리할 수 있게 한다.

#### 구현 요구사항

- 사용자는 학습 기록 상세의 `생성된 문제` 탭에서 현재 활성 문제 목록을 볼 수 있다.
- 각 항목은 문제 문장, 생성 시각, 삭제 버튼을 보여준다.
- 사용자는 개별 문제를 삭제할 수 있다.
- 문제 삭제 후 Queue는 다시 정렬된다.

#### 제약

- 현재 UI는 정답을 이 탭에 노출하지 않는다.
- 문제 수정, 재생성, 수동 순서 변경 기능은 없다.

### 6.6 Queue 현황

#### 목적

사용자가 현재 전역 Queue의 진행 상황과 다음 문제를 미리 확인하게 한다.

#### 구현 요구사항

- 시스템은 Queue 현황 화면에서 현재 전체 문제 수, 완료 수, 진행률을 보여준다.
- 시스템은 현재 Queue의 다음 문제를 미리 보여준다.
- 사용자는 `문제 풀기` 버튼으로 풀이 화면에 진입할 수 있다.
- Queue가 비어 있으면 빈 상태 안내와 홈 이동 버튼을 보여준다.

#### 제약

- Queue는 학습 기록별이 아니라 앱 전체 문제를 하나로 합친 전역 Queue다.

### 6.7 문제 풀이

#### 목적

사용자가 시스템이 정한 현재 문제를 하나씩 풀도록 한다.

#### 구현 요구사항

- 문제 풀이 화면은 진입 시 현재 문제를 조회한다.
- 시스템은 현재 문제의 학습 기록 제목, 문제 문장, 난이도, 타이머를 보여준다.
- 사용자는 서술형 답변을 입력할 수 있다.
- 공란 제출은 허용되지 않는다.
- 제출 시 시스템은 다음 데이터를 저장한다.
  - 현재 문제 ID
  - 사용자가 입력한 답변
  - 풀이 시간(초)
- 제출 후 시스템은 다음 중 하나를 수행한다.
  - 같은 사이클의 다음 문제로 이동
  - 특정 학습 기록 완주 요약 모달 표시
  - 전체 사이클 완료 전환 연출 실행

#### 풀이 시간 규칙

- 프론트는 문제가 로드되면 타이머를 0초부터 증가시킨다.
- 현재 구현에서 Zustand 스토어는 실제 경과 시간 계산보다 시작/정지 흔적 저장 용도로만 사용되고, 화면 표시는 페이지 로컬 상태로 관리된다.

#### 제약

- 자동 채점하지 않는다.
- 제출 직후 정답을 즉시 보여주지 않는다.

### 6.8 학습 기록 단위 완주 요약

#### 목적

사용자가 특정 학습 기록의 모든 활성 문제를 현재 사이클에서 풀었을 때, 원문과 답변을 함께 비교하며 복습 품질을 판단하게 한다.

#### 구현 요구사항

- 시스템은 현재 제출한 문제의 학습 기록에 속한 활성 문제들이 모두 현재 사이클에서 시도되었는지 확인한다.
- 모두 시도되었으면 `CompletionQuizzesModal`을 연다.
- 모달 왼쪽 패널은 문제별 비교를 제공한다.
  - 문제 문장
  - 내 답변
  - 정답
  - 소요 시간
  - `별로에요` 표시 토글
- 모달 오른쪽 패널은 학습 기록 원문 전체를 보여준다.
- 사용자는 이번 기록 난이도를 다음 셋 중 하나로 반드시 평가해야 한다.
  - 어려워요
  - 적당해요
  - 쉬워요
- 품질 평가는 선택이며, 여러 문제에 `별로에요`를 표시할 수 있다.
- 난이도 평가가 저장되어야만 `계속 풀기` 버튼이 활성화된다.

#### 데이터 규칙

- 평가 저장은 현재 사이클의 해당 학습 기록 문제들에 대한 `quiz_attempt` 레코드 전체를 업데이트한다.
- 선택된 난이도 평가는 각 시도 레코드의 `selfEvaluation`에 기록된다.
- `별로에요`로 표시된 문제는 각 시도 레코드의 `problemFeedback = DISLIKED`로 저장된다.

#### 제약

- 평가 저장에 실패하면 사용자는 계속 진행할 수 없다.
- 개별 문제별 난이도 평가는 없다. 학습 기록 완주 시점에 해당 기록 내 문제들에 동일한 난이도 평가가 일괄 반영된다.

### 6.9 전체 사이클 완료와 이관

#### 목적

Queue 전체를 한 바퀴 다 돌았을 때 사이클 종료를 명확히 보여주고, 현재 사이클 데이터를 다음 루프 시작 전에 정리한다.

#### 구현 요구사항

- 시스템은 제출 후 `completedCount + 1 >= totalCount`이면 현재 사이클 완료로 본다.
- 사이클 완료 시 다음 문제 포인터를 설정하면서 `cycleJustCompleted = true`로 표시한다.
- 다음 문제 조회 전에 시스템은 이전 사이클의 현재 시도들을 히스토리 테이블로 이관한다.
- 이관 후 현재 시도 테이블은 비워진다.
- 프론트는 사이클 완료 시 시네마틱 오버레이를 보여준다.
- 오버레이는 최소 약 6초 이상 유지되며 다음 단계를 순서대로 노출한다.
  - cycle_complete
  - analyzing
  - regenerating
  - finalizing

#### 내부 구현

- Queue 상태 갱신은 일부를 DB 원자적 UPDATE로 처리한다.
- 현재 사이클 이관은 `getCurrentQuiz()` 호출 직전에 수행된다.

### 6.10 적응형 난이도

#### 목적

사용자의 체감 난이도를 다음 사이클 문제 수준에 반영한다.

#### 구현 요구사항

- 이전 사이클의 시도 이관 직전에 시스템은 각 시도의 `selfEvaluation`을 읽는다.
- 현재 Queue에 활성 상태로 남아 있는 문제에 대해 다음 규칙을 적용한다.
  - `TOO_EASY`면 난이도 +1
  - `TOO_HARD`면 난이도 -1
  - `OK`면 유지
- 목표 난이도 문제 변형이 이미 존재하면 재사용한다.
- 없으면 Gemini를 호출해 같은 학습 내용 기반의 변형 문제를 새로 만든다.
- 새 변형 문제는 기본적으로 비활성이며, 교체가 필요할 때만 기존 문제를 대체해 활성 Queue에 들어간다.

#### 제약

- 삭제된 학습 기록에 속한 문제는 변형 문제를 새로 만들 수 없다.
- Gemini 변형 생성 실패 시 원본 문제를 유지한다.

### 6.11 풀이 기록

#### 목적

사용자가 현재 사이클과 과거 사이클을 모두 포함한 풀이 이력을 확인하게 한다.

#### 구현 요구사항

- 시스템은 `/history`에서 현재 사이클과 과거 사이클 기록을 함께 보여준다.
- 기록은 사이클 단위로 그룹핑된다.
- 현재 사이클은 `진행 중` 배지와 함께 최상단에 표시된다.
- 과거 사이클은 `migratedAt` 분 단위 기준으로 묶여 표시된다.
- 각 풀이 항목은 다음 정보를 보여준다.
  - 학습 기록 제목
  - 문제
  - 난이도
  - 난이도 평가
  - 품질 평가
  - 소요 시간
  - 내 답변
  - 정답
- 학습 기록 상세의 `풀이 기록` 탭은 해당 기록에 속한 문제만 필터링해 문제별 집계형으로 보여준다.
- 문제별 집계에서는 다음 정보를 보여준다.
  - 문제 문장
  - 최신 난이도
  - 최신 난이도 평가
  - 최신 품질 평가
  - 누적 풀이 횟수
  - 평균 풀이 시간
  - 최근 풀이 날짜
  - 최신 내 답변
  - 정답

### 6.12 삭제와 데이터 보존

#### 학습 기록 삭제

- 학습 기록은 삭제되지만 관련 문제 행은 보존될 수 있다.
- 관련 문제는 Queue에서 제외된다.
- 관련 풀이 기록은 남는다.
- 남은 기록은 `(삭제된 기록)`이라는 제목으로 집계/표시될 수 있다.

#### 문제 삭제

- 개별 문제 삭제 시 현재 사이클 시도와 과거 히스토리도 함께 삭제된다.
- 삭제 후 Queue 순서는 다시 정렬된다.

## 7. 상세 사용자 스토리

### Epic A. 학습 재료 준비

#### US-A1. 새 학습 기록 작성

학습자로서, 나는 공부한 내용을 제목과 본문으로 저장하고 싶다. 그래야 이후 AI 문제 생성의 재료로 사용할 수 있다.

- 사전조건: 앱에 로그인 개념은 없고 누구나 바로 진입 가능하다.
- 트리거: 사용자가 `새 학습 기록 작성` 버튼을 클릭한다.
- 기본 흐름:
  1. 시스템은 작성 페이지를 연다.
  2. 사용자는 제목과 내용을 입력한다.
  3. 사용자가 저장을 누르면 시스템은 입력값 앞뒤 공백을 제거한 뒤 저장 요청을 보낸다.
  4. 저장 성공 시 학습 기록 목록으로 이동한다.
- 예외 흐름:
  - 제목 또는 내용이 비어 있으면 저장되지 않는다.
  - 저장 실패 시 오류 메시지를 보여준다.
- 완료 기준:
  - 목록에 새 기록이 나타난다.
  - 상세 페이지 진입 가능한 ID가 생성된다.

#### US-A2. 학습 기록 목록 확인

학습자로서, 나는 지금까지 쌓은 기록을 한 번에 보고 싶다. 그래야 어떤 재료가 이미 준비됐는지 알 수 있다.

- 시스템은 목록 진입 시 전체 학습 기록을 로드한다.
- 시스템은 각 기록에 대해 제목, 본문 일부, 생성일, 활성 문제 수를 보여준다.
- 사용자는 아무 카드나 눌러 상세 화면으로 이동할 수 있다.
- 기록이 없으면 빈 상태 메시지와 작성 CTA를 보여준다.

#### US-A3. 학습 기록 상세 확인

학습자로서, 나는 저장한 원문을 다시 읽고 그 기록에 연결된 문제/설정/풀이 이력을 함께 보고 싶다.

- 시스템은 단건 조회 성공 시 제목, 작성일, 탭 UI를 보여준다.
- 사용자는 `학습 내용`, `생성된 문제`, `풀이 기록`, `문제 생성 설정`을 오갈 수 있다.
- 존재하지 않는 ID이면 오류 안내 후 목록으로 돌아갈 수 있다.

#### US-A4. 학습 기록 삭제

학습자로서, 나는 더 이상 유지할 필요가 없는 기록을 제거하고 싶다. 하지만 기존 풀이 이력은 통계용으로 남아도 된다.

- 사용자는 상세 화면에서 삭제 버튼을 누른다.
- 시스템은 확인 모달을 띄운다.
- 시스템은 삭제 영향 범위를 설명한다.
  - 연관 문제는 더 이상 출제되지 않음
  - 풀이 기록은 보존됨
- 삭제 성공 시 목록으로 이동한다.
- 삭제 실패 시 오류 모달을 보여준다.

### Epic B. AI 문제 세트 구성

#### US-B1. 문제 생성 설정 생성

학습자로서, 나는 같은 학습 기록에서도 서로 다른 관점의 문제 세트를 만들고 싶다.

- 사용자는 설명과 문제 수를 입력한다.
- 시스템은 해당 학습 기록 아래 설정을 생성한다.
- 생성 후 설정 목록이 새로고침된다.
- 설명이 비어 있으면 정보 모달로 입력 필요를 알린다.

#### US-B2. 여러 설정 관리

학습자로서, 나는 "핵심 개념 정리", "키워드 회상", "응용 질문"처럼 설정을 여러 개 유지하고 싶다.

- 한 학습 기록에 여러 설정을 저장할 수 있다.
- 시스템은 각 설정에 대해 설명과 문제 수를 표시한다.
- 사용자는 필요 없는 설정을 삭제할 수 있다.

#### US-B3. AI 문제 생성 실행

학습자로서, 나는 저장한 설정을 실제 문제로 변환하고 싶다.

- 사용자는 설정 카드의 `문제 생성` 버튼을 누른다.
- 시스템은 해당 설정과 학습 기록 내용을 Gemini에 전달한다.
- 응답을 문제/정답 쌍으로 저장한다.
- 저장 후 전체 Queue 순서를 다시 정렬한다.
- 생성 성공 시 성공 모달을 띄우고 문제 탭 이동을 유도한다.
- 생성 실패 시 오류 모달을 띄운다.

#### US-B4. 생성된 문제 확인 및 정리

학습자로서, 나는 생성된 문제를 훑어보고 불필요한 문제는 제거하고 싶다.

- 사용자는 `생성된 문제` 탭에서 문제 목록을 본다.
- 각 문제는 질문 텍스트와 생성 시각을 보여준다.
- 사용자는 개별 문제를 삭제할 수 있다.
- 삭제 시 목록이 즉시 새로고침된다.

### Epic C. Queue 기반 반복 풀이

#### US-C1. Queue 상태 확인

학습자로서, 나는 지금 전체 복습 루프에서 어디까지 왔는지 알고 싶다.

- 사용자는 Queue 현황 화면에서 완료 수, 전체 수, 퍼센트를 본다.
- 사용자는 다음 문제 미리보기를 본다.
- Queue가 비어 있으면 문제 생성 유도 메시지를 본다.

#### US-C2. 현재 문제 풀이 시작

학습자로서, 나는 시스템이 정해준 현재 문제를 바로 풀고 싶다.

- 문제 풀이 화면은 진입 시 현재 문제를 로드한다.
- 시스템은 문제, 학습 기록 제목, 난이도, 타이머를 보여준다.
- 사용자는 답변 입력창에 자유롭게 답을 작성한다.

#### US-C3. 답안 제출

학습자로서, 나는 답안을 제출하고 바로 다음 문제 흐름으로 이어지고 싶다.

- 사용자는 공란이 아닌 답안을 입력하고 제출한다.
- 시스템은 풀이 시간을 함께 전송한다.
- 시스템은 시도를 현재 사이클 테이블에 저장한다.
- 시스템은 다음 문제 포인터를 계산해 Queue 상태를 갱신한다.

#### US-C4. 빈 답변 방지

학습자로서, 나는 실수로 빈 답안을 제출하지 않길 원한다.

- 빈 문자열 또는 공백만 입력된 경우 시스템은 알림 모달을 띄우고 제출하지 않는다.

#### US-C5. Queue가 비었을 때 안내

학습자로서, 나는 풀 문제가 없을 때 왜 비어 있는지 바로 이해하고 다음 행동으로 이동하고 싶다.

- 현재 문제가 없으면 빈 상태 안내를 표시한다.
- 사용자는 홈으로 돌아가 학습 기록 작성 또는 문제 생성 흐름으로 이동할 수 있다.

### Epic D. 기록 단위 완주와 복습 검증

#### US-D1. 특정 학습 기록 완주 감지

학습자로서, 나는 한 학습 기록에 속한 모든 문제를 현재 사이클에서 풀었을 때 그 사실을 즉시 알고 싶다.

- 제출된 문제가 속한 학습 기록의 활성 문제 전체가 현재 사이클에서 시도되었는지 검사한다.
- 모두 시도되었다면 완주 요약 모달을 연다.

#### US-D2. 내 답과 정답 비교

학습자로서, 나는 방금까지 회상한 답과 실제 정답을 나란히 보고 싶다.

- 모달은 문제별로 내 답변과 정답을 비교해 보여준다.
- 사용자는 아코디언 형태로 문제를 펼쳐볼 수 있다.
- 각 문제의 풀이 시간도 확인할 수 있다.

#### US-D3. 원문 재검토

학습자로서, 나는 문제만 보는 것이 아니라 원래 학습한 내용을 다시 읽고 싶다.

- 완주 모달 우측에는 학습 기록 원문이 그대로 노출된다.
- 사용자는 문제 비교와 원문 검토를 한 화면에서 수행할 수 있다.

#### US-D4. 난이도 평가 입력

학습자로서, 나는 이 기록의 문제가 나에게 쉬웠는지, 적절했는지, 어려웠는지 시스템에 알려주고 싶다.

- 사용자는 세 가지 난이도 평가 중 하나를 선택한다.
- 선택 즉시 시스템은 현재 기록의 현재 사이클 시도들에 평가를 저장한다.
- 저장 성공 전까지는 계속 진행할 수 없다.

#### US-D5. 품질이 낮은 문제 표시

학습자로서, 나는 품질이 별로였던 문제를 표시하고 다음 개선에 반영되길 원한다.

- 사용자는 각 문제의 `별로에요` 버튼을 토글할 수 있다.
- 난이도 평가가 이미 저장된 상태라면 토글할 때마다 다시 저장된다.
- 난이도 평가 전에는 표시 상태만 바뀌고, 실제 저장은 난이도 평가 후에 함께 이뤄진다.

#### US-D6. 완주 후 계속 풀이

학습자로서, 나는 완주 검토를 마친 뒤 다음 문제 흐름으로 자연스럽게 돌아가고 싶다.

- 난이도 평가 저장이 끝나면 `계속 풀기` 버튼이 활성화된다.
- 사용자가 누르면 다음 문제를 로드하거나, 전체 사이클 완료였다면 사이클 전환 오버레이를 시작한다.

### Epic E. 사이클 완료와 적응형 루프

#### US-E1. 전체 사이클 완료 인지

학습자로서, 나는 모든 문제를 한 바퀴 다 풀었을 때 명확한 종료감을 느끼고 싶다.

- 마지막 문제 제출 후 시스템은 `isCycleComplete = true`를 반환한다.
- 프론트는 일반 다음 문제 이동 대신 전환 오버레이를 띄운다.

#### US-E2. 사이클 전환 연출 경험

학습자로서, 나는 다음 루프로 넘어가는 동안 시스템이 재정비 중이라는 감각을 얻고 싶다.

- 오버레이는 분석, 재배치, 안정화 메시지를 순차적으로 보여준다.
- 최소 시간 경과와 백엔드 준비 완료가 모두 충족되어야 닫힌다.

#### US-E3. 이전 사이클 이력 보존

학습자로서, 나는 매 사이클의 풀이 흔적이 사라지지 않고 히스토리로 누적되길 원한다.

- 시스템은 현재 사이클 시도 전체를 히스토리 테이블로 복사한다.
- 이후 현재 사이클 테이블은 비워 다음 루프를 시작한다.

#### US-E4. 난이도 적응

학습자로서, 나는 쉬웠던 문제는 조금 더 어렵게, 어려웠던 문제는 조금 더 쉽게 바뀌길 원한다.

- 시스템은 각 문제의 난이도 평가를 읽어 다음 사이클 목표 난이도를 결정한다.
- 필요한 경우 같은 주제의 변형 문제를 생성하거나 기존 변형을 재사용한다.
- 교체가 발생하면 활성 Queue는 다시 정렬된다.

### Epic F. 학습 통계와 회고

#### US-F1. 대시보드로 전체 상황 파악

학습자로서, 나는 오늘 얼마나 공부했는지와 전체 복습 루프 상태를 한눈에 알고 싶다.

- 대시보드는 누적 및 당일 지표를 한 화면에 제공한다.
- 사용자는 다음 문제, 현재 진행률, 최근 활동량을 동시에 확인할 수 있다.

#### US-F2. 많이 푼 기록 확인

학습자로서, 나는 어떤 학습 기록을 반복적으로 많이 풀었는지 알고 싶다.

- 대시보드는 상위 학습 기록을 풀이 횟수와 학습 시간 기준으로 보여준다.
- 사용자는 항목을 눌러 해당 기록 상세로 이동한다.

#### US-F3. 전체 풀이 기록 회고

학습자로서, 나는 현재 사이클과 과거 사이클을 구분해서 되짚어보고 싶다.

- 풀이 기록 화면은 현재 사이클과 완료된 사이클을 분리해 보여준다.
- 각 사이클을 펼쳐 개별 문제, 답, 정답, 난이도, 평가를 확인할 수 있다.

#### US-F4. 기록별 문제 성과 검토

학습자로서, 나는 특정 학습 기록에서 어떤 문제가 반복적으로 어려웠는지 알고 싶다.

- 학습 기록 상세의 `풀이 기록` 탭은 문제별 집계형 정보를 보여준다.
- 사용자는 최신 답과 정답, 평균 시간, 최신 평가를 문제 단위로 확인할 수 있다.

## 9. 주요 비즈니스 규칙

### 9.1 Queue 규칙

- Queue는 앱 전체의 활성 문제를 하나로 합친다.
- `queueOrder`는 활성 문제 기준으로 1부터 재정렬된다.
- Queue 변경이 일어나면 현재 포인터는 첫 문제로 재설정될 수 있다.

### 9.2 사이클 규칙

- `completedCount`는 현재 사이클에서 몇 문제를 풀었는지 뜻한다.
- 마지막 문제 제출 시 다음 사이클의 첫 문제를 가리키도록 상태를 갱신한다.
- 사이클 완료 직후 다음 문제 조회 전에 현재 시도들이 히스토리로 이동한다.

### 9.3 완주 규칙

- 특정 학습 기록의 활성 문제 전부가 현재 사이클의 `quiz_attempt`에 존재하면 그 기록은 완주로 간주된다.

### 9.4 난이도 규칙

- 기본 난이도: 5
- 난이도 범위: 1~10
- 쉬웠다: +1
- 어려웠다: -1
- 적당했다: 유지

### 9.5 삭제 규칙

- 학습 기록 삭제는 풀이 이력을 지우지 않는다.
- 개별 문제 삭제는 해당 문제의 현재/과거 시도를 함께 지운다.

## 13. 현재 구현상 제약 및 관찰 사항

- 학습 기록, 문제, 설정의 수정 기능이 없다.
- 문제 풀이는 전부 전역 Queue 기준이며, 학습 기록별 개별 학습 모드는 없다.
- 정답 자동 판정이 없으므로 사용자는 완주 모달 또는 이력 화면에서 스스로 검토해야 한다.
- 완주 후 계속 진행하려면 난이도 평가 저장이 필수다.
- Queue 초기화 API는 구현되어 있으나 현재 FE에서 직접 노출되지 않는다.
- `StudyLogCompleteModal` 컴포넌트는 코드에 남아 있으나 현재 문제 풀이 흐름에서는 사용되지 않는다.
- 테스트 태스크는 존재하지만 실제 테스트 소스는 없다.

## 14. 요약

Recall Loop의 현재 구현은 "학습 기록 저장 → AI 문제 생성 → 전역 Queue 반복 풀이 → 기록 단위 완주 검토 → 사이클 단위 히스토리 누적 → 평가 기반 난이도 적응"이라는 폐쇄 루프를 완성한 상태다.

즉, 이 제품은 단순 메모 앱도, 단순 퀴즈 앱도 아니라 다음의 다섯 축으로 정의된다.

- 학습 원문 저장
- AI 기반 문제화
- 전역 Queue 반복 복습
- 완주 시점의 회고/피드백
- 사이클 기반 이력 누적과 적응형 난이도 조정
